## Overview
This `docs` folder serves as the central repository for all developer-facing documentation for this project. The documentation contained within is generated directly from the source code and configuration files, providing an accurate, up-to-date reference for the development team.

The primary purpose of this directory is to document the critical, non-business-logic components of the application's architecture. This includes:
*   **Continuous Integration (`.circleci`):** Detailed breakdown of the CI/CD pipeline, jobs, and workflows.
*   **Testing (`cypress`):** Comprehensive documentation of the entire testing suite, including E2E, API, and component tests, as well as support commands and utilities.
*   **Build & Development Configuration:** Explanations of core configuration files like `vite.config.ts` and `cypress.config.ts`.

By maintaining this repository, we aim to streamline developer onboarding, improve code maintainability, and provide clear insight into the project's operational backbone.

## Usage
This documentation is intended to be used as a reference. There is no installation or instantiation required. Developers should navigate this directory to gain a deeper understanding of the project's structure and processes.

**To find information:**
*   **To understand the CI/CD pipeline:** Navigate to the `docs/.circleci/` directory and review the documentation for `config.yml`.
*   **To explore the testing framework:** The `docs/cypress/` directory contains all test-related documentation. You can drill down into specific test types (`api`, `ui`), support commands, or configuration.
*   **To review project configuration:** Top-level files like `vite.config.ts.md` provide insight into the project's build and development server setup.

The structure of this `docs` folder intentionally mirrors the project's root directory, making it intuitive to find the documentation for a specific file or feature.

*   **Generated Content:** All markdown files (`.md`) in this directory are generated and should be treated as read-only. Any necessary changes should be made in the corresponding source file, after which the documentation can be regenerated.
