## Overview

This directory serves as the root for the project's entire automated testing suite, powered by Cypress. It provides a comprehensive framework for ensuring application quality through End-to-End (E2E) UI testing, component testing, and direct API validation.

The primary purpose of this folder is to create a robust, maintainable, and scalable testing environment. It is designed to simulate user interactions, verify application logic from the frontend to the backend, and catch regressions before they reach production. This self-contained ecosystem includes all test specifications, reusable commands, configuration files, and support utilities required to automate the quality assurance process.

## Usage

The tests within this framework can be executed in two main ways: through the interactive Test Runner for development and debugging, or headlessly via the command line for CI/CD integration.

**1. Interactive Mode (for development):**

To open the Cypress Test Runner, which allows you to see tests run in a live browser and use debugging tools, execute the following command from the project's root directory:

```bash
npx cypress open
```

**2. Headless Mode (for CI/CD):**

To run all tests headlessly in the terminal (as it would be done in a CI pipeline), use the following command:

```bash
npx cypress run
```

You can also run tests for a specific file by using the `--spec` flag:

```bash
npx cypress run --spec "cypress/tests/ui/auth.spec.ts"
```

## Useful details

The `cypress` directory is organized to separate concerns, promoting code reuse and clarity. Understanding this structure is key to effectively contributing to the testing suite.

*   `cypress.config.ts`: The master configuration file for Cypress. It defines global settings such as base URLs, default timeouts, viewport dimensions, and environment-specific variables. All test runs are governed by the settings in this file.

*   **`/tests`**: This subfolder is the core of the test suite, containing all the actual test specifications (`.spec.ts` files). It is further organized by testing type:
    *   **`/api`**: Contains tests that directly call and validate the application's API endpoints, bypassing the UI. These are crucial for fast, targeted validation of backend logic.
    *   **`/ui`**: Holds the primary end-to-end tests that simulate user journeys through the application's graphical interface.
    *   **`/ui-auth-providers`**: Specialized UI tests that focus exclusively on verifying authentication flows with various third-party identity providers (e.g., Auth0, Okta, Cognito).
    *   **`/demo`**: Includes example or exploratory tests, often used for demonstrating Cypress features or for proof-of-concept work.

*   **`/support`**: This folder contains reusable code and configurations that are globally available across all test files. Its purpose is to keep tests clean, readable, and DRY (Don't Repeat Yourself).
    *   `e2e.ts`: The main entry point for support files in E2E testing. It imports custom commands and other global setup logic.
    *   `commands.ts`: Defines custom Cypress commands (e.g., `cy.login()`, `cy.createBankAccount()`) that abstract complex or repetitive sequences of actions into single, readable functions.
    *   **`/auth-provider-commands`**: A modular collection of custom commands specifically for handling authentication with different identity providers, keeping the main `commands.ts` file clean.
    *   `utils.ts`: A collection of general-purpose helper functions that do not directly interact with the Cypress `cy` object but are useful for test data generation, calculations, or other logic.

*   `global.d.ts`: A TypeScript declaration file. It provides type definitions for the custom commands created in `support/commands.ts`, enabling type safety and intelligent code completion in modern IDEs.
