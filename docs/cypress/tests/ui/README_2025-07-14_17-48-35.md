## Overview

The `ui` folder contains Cypress end-to-end test suites that comprehensively verify all major user interface (UI) workflows within the application. These tests replicate real-world user behavior across core features such as authentication, onboarding, bank account management, transactions, notifications, feed interactions, and user profile settings. By using database seeding, GraphQL and HTTP API intercepts, and extensive DOM assertions, the test suites ensure the frontend application remains robust, reliable, and user-friendly across common usage scenarios and edge cases.

Within a software project, these UI tests provide critical regression coverage and act as living documentation of expected application behavior. They help developers, QA engineers, and product stakeholders verify that new releases or changes do not break critical user flows and that the UX remains consistent across devices and supported authentication schemes.

## Usage

1. **Setup:**
   - Ensure your project has Cypress configured.
   - The application under test and its backend must be available and configured for the appropriate test environment.
   - Some tests intercept or interact with GraphQL APIs and may rely on specific database states—seed data as specified by the individual test suites.

2. **Folder Structure:**

   - Each `.spec.ts.md` file represents a Cypress test suite for a specific UI area:
     - `auth.spec.ts.md` - Authentication (sign-up, login, onboarding, form validation)
     - `bankaccounts.spec.ts.md` - Bank account CRUD operations and validation
     - `new-transaction.spec.ts.md` - Creating new transactions, error validation, and list appearance
     - `transaction-view.spec.ts.md` - Transaction detail page, like/comment, request response flows
     - `transaction-feeds.spec.ts.md` - Transaction feed filters, pagination, and list ownership rules
     - `notifications.spec.ts.md` - UI notifications, interaction, badge counts, empty state handling
     - `user-settings.spec.ts.md` - Profile and settings update, form validation, and submission feedback

3. **Running Tests:**
   - To run tests locally:  
     ```bash
     npx cypress open
     ```
     or headless:
     ```bash
     npx cypress run --spec "cypress/tests/ui/*.spec.ts"
     ```
   - Tests are designed to be standalone and can be executed independently or as part of a continuous integration pipeline.

4. **Interacting with tests:**
   - Tests use custom Cypress commands (`cy.login`, `cy.getBySel`, etc.), API/database seeded fixtures, and intercept network requests for deterministic results.
   - Responsive and multi-device UI logic is exercised—tests may adapt steps for mobile and desktop UIs.

## Useful details

- **Authentication Variants:** Some flows require Auth0, Google, Cognito, or Okta configurations. These are tested in `ui-auth-providers` but may be referenced for modularity.
- **Data seeding and intercepts:** Almost every suite seeds the database or intercepts GraphQL/REST calls to control responses and ensure predictable UI states. Be sure your backend supports these hooks.
- **Accessibility and Responsive Checks:** Several tests check the UI both on desktop and mobile, validating navigation drawers and sidebars where applicable.
- **Visual Snapshots:** Visual regression checks using `cy.visualSnapshot` provide confidence that UI changes do not affect layout or content presentation.
- **Error and Edge Cases:** Tests intentionally provoke validation and error states (invalid inputs, empty data sets, failed logins, etc.) to ensure UX resilience.
- **Custom Cypress Commands:** The tests employ helper commands like `cy.loginByXstate`, `cy.switchUserByXstate`, `cy.getBySelLike`, and more for concise, reusable logic.
- **Test Data:** Some tests rely on stubbed fixtures (e.g., "public-transactions.json") and named intercepts (`@updateTransaction`, `@usersSearch`). Keep these fixtures current with backend expectations.

**Best Practice:** When adding new UI features, create a corresponding Cypress `.spec.ts` in this folder to maintain end-to-end coverage. Review and update fixtures/intercepts as backend APIs evolve.

**For more details:** Review the headers and inline step descriptions in each `.spec.ts.md` file, which provide scenario-specific logic, setup requirements, and assertions.
