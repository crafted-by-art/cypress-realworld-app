## Overview
This folder contains the Cypress end-to-end (E2E) test suites for the core user interface (UI) and application functionality. These tests validate the primary user-facing features that are available after a user has successfully authenticated.

The primary purpose of these tests is to simulate real user workflows and interactions to ensure the application behaves as expected. This includes everything from data creation and management (transactions, bank accounts) to social interactions (notifications, comments) and user profile settings. By covering these critical paths, this test suite acts as a safety net against regressions and bugs in the core product experience.

The suites within this directory test the following key areas:
*   **Authentication:** Native user sign-up, login, and session management.
*   **Bank Account Management:** Creation, validation, and listing of bank accounts.
*   **Transactions:** Creating new payments/requests, viewing transaction feeds (with filtering and infinite scroll), and interacting with individual transaction details (liking, commenting, accepting/rejecting).
*   **User Profile:** Viewing and updating user settings with proper form validation.
*   **Notifications:** Verifying that user actions trigger appropriate notifications and that the notification list renders correctly.

## Usage
These tests are designed to be run using the Cypress Test Runner. Ensure that the project dependencies are installed and the application (both frontend and backend) is running and accessible at the configured `baseUrl` in `cypress.config.ts`.

**1. Open the Cypress Test Runner:**
To run tests interactively, which is recommended for development and debugging, use the `open` command. This will launch the Cypress UI, where you can select and run specific test files from the `ui` folder.

```bash
npx cypress open
```

**2. Run Tests Headlessly:**
To run the entire suite or a specific file in a headless browser (ideal for CI/CD environments), use the `run` command.

```bash
# Run all tests in the ui folder
npx cypress run --spec "cypress/e2e/ui/*.spec.ts"

# Run a single test file
npx cypress run --spec "cypress/e2e/ui/new-transaction.spec.ts"
```

## Useful details
The tests in this directory leverage several powerful Cypress patterns and custom commands to ensure they are robust, reliable, and maintainable.

*   **Database Seeding:** Most test suites use a `beforeEach` hook to execute `cy.task("db:seed")`. This practice ensures that each test runs against a clean, predictable database state, which is fundamental for test isolation and preventing flaky results.

*   **API Interception:** `cy.intercept()` is used extensively to manage network requests. This allows tests to:
    *   Wait for critical API calls to complete before making assertions (`cy.wait('@alias')`).
    *   Verify the payloads of requests and responses.
    *   Mock API responses to test edge cases, such as empty states or server errors, without needing to manipulate the backend.

*   **Programmatic Login:** To improve speed and reduce dependency on the login UI, tests primarily use custom commands like `cy.loginByXstate` for programmatic authentication. This isolates the test's focus to the specific feature being validated.

*   **Custom Commands:** The project utilizes a library of custom commands (e.g., `cy.switchUserByXstate`, `cy.database`) to abstract complex or repetitive actions, making the test code cleaner and more readable.

*   **Visual Regression Testing:** Some suites, such as `user-settings.spec.ts`, incorporate `cy.visualSnapshot`. This helps catch unintended UI changes by comparing screenshots of key components against a baseline, ensuring visual consistency across releases.
