## Overview
This directory contains the suite of Cypress integration tests dedicated to validating the application's backend API. These tests are crucial for ensuring the stability, reliability, and correctness of the server-side logic.

Unlike end-to-end tests that interact with the UI, these API tests communicate directly with the application's endpoints (both REST and GraphQL). They verify the complete request-response cycle, including data validation, business logic execution, database interactions, and authentication/authorization. By isolating the backend, we can quickly pinpoint and diagnose issues without the overhead of the browser UI, making them an essential part of our CI/CD pipeline and overall quality assurance strategy.

## Usage
These tests are designed to be run against a running instance of the application's backend. They follow a consistent structure, making it straightforward to add new tests for new or existing endpoints.

### Running the Tests
To execute all API tests in this directory, use the following Cypress command from the project root:
```bash
npx cypress run --spec "cypress/tests/api/**/*.spec.ts"
```

### Writing a New API Test
When adding a new test file, adhere to the established pattern for consistency and reliability:

1.  **Create a new file:** Name it according to the API resource being tested, e.g., `api-payments.spec.ts`.
2.  **Use the `beforeEach` hook:** Set up a clean and authenticated state for each test. This typically involves:
    *   **Seeding the database:** `cy.task("db:seed")` to ensure a predictable data state.
    *   **Authenticating a user:** `cy.loginByApi()` to obtain valid authentication credentials for making protected requests.
    *   **Fetching test data:** Use `cy.database()` or `cy.request()` to retrieve necessary entity IDs (e.g., user IDs, transaction IDs) for the test.
3.  **Write the test case:**
    *   Use `cy.request()` to perform the HTTP request (`GET`, `POST`, `PATCH`, `DELETE`, etc.) against the target endpoint.
    *   Chain `.then()` to the request to access the response.
    *   Use `expect()` assertions to validate the response status code, headers, and body content.

```typescript
// Example of a typical test structure
describe('Example API', () => {
  beforeEach(() => {
    cy.task("db:seed");
    cy.loginByApi();
  });

  it('should retrieve a list of items', () => {
    cy.request('GET', '/api/items').then((response) => {
      expect(response.status).to.eq(200);
      expect(response.body.results).to.be.an('array');
    });
  });
});
```

## Useful details
This section outlines key components and the specific purpose of each test file within this directory.

### Core Test Utilities
Our API tests rely on several custom Cypress commands to streamline setup and improve readability:
*   `cy.loginByApi()`: A custom command that programmatically authenticates a user and stores the session information, bypassing the UI login flow.
*   `cy.task("db:seed")`: A command that connects to the test database and resets it to a known, default state.
*   `cy.database()`: A utility command for querying the database directly to fetch or verify data needed for test setup or assertions.

### Component Test Suites
Below is a breakdown of the test files and the API resources they cover:

*   **`api-bankaccounts.spec.ts`**: Validates CRUD operations for user bank accounts via both REST (`/bankAccounts`) and GraphQL endpoints.
*   **`api-banktransfers.spec.ts`**: Ensures the `/bankTransfers` endpoint correctly lists transfers for an authenticated user.
*   **`api-comments.spec.ts`**: Tests the creation and retrieval of comments associated with a specific transaction.
*   **`api-contacts.spec.ts`**: Covers the management of user contacts, including creation, retrieval, deletion, and validation of input.
*   **`api-likes.spec.ts`**: Verifies the creation (`POST`) and retrieval (`GET`) of "likes" for transactions.
*   **`api-notifications.spec.ts`**: Validates the functionality of the notifications system, including fetching, bulk creation, and marking notifications as read.
*   **`api-testdata.spec.ts`**: A utility test suite that validates the `/testData/:entity` endpoints, which are used to fetch seed data for tests.
*   **`api-transactions.spec.ts`**: A comprehensive suite for the core Transactions API, testing creation, updates, and retrieval of various transaction-related data.
*   **`api-users.spec.ts`**: Tests the Users API, covering user creation, retrieval (by ID, list), profile management, and updates.
