## Overview

This folder contains Cypress test suites focused on validating the API layer functionalities of a financial web application. Its primary purpose is to ensure the correctness, reliability, and security of the REST and GraphQL API endpoints that underpin core product features such as bank account management, user authentication, transactions, contacts, notifications, likes, comments, and test data handling.

By interacting directly with the backend APIs, these tests emulate real-world usage, verifying both positive and negative scenarios, data consistency, error handling, and user access controls. These suites are crucial for maintaining the integrity of the application, enabling rapid regression testing as APIs evolve, and catching issues early in the development lifecycle.

## Usage

1. **Pre-requisites**
   - Ensure [Cypress](https://www.cypress.io/) is installed in your project.
   - The backend server (and relevant test DB) should be up and seeded, accessible from the Cypress environment.
   - Cypress custom commands (`cy.database`, `cy.task`, `cy.loginByApi`, etc.) must be properly defined in your Cypress support files.

2. **Executing the Tests**

   - From your project root, run the following command to execute all API tests in this folder:
     ```
     npx cypress run --spec cypress/tests/api/*.spec.ts
     ```
   - To run an individual test suite, specify its filename:
     ```
     npx cypress run --spec cypress/tests/api/api-bankaccounts.spec.ts
     ```

3. **Test Suite Organization**

   Each `.spec.ts` file targets a logical domain or entity of the API:
   - **api-bankaccounts.spec.ts**: Tests CRUD operations for bank accounts via REST/GraphQL.
   - **api-banktransfers.spec.ts**: Verifies authenticated GET requests for bank transfers.
   - **api-comments.spec.ts**: Ensures comments can be fetched and created on transactions.
   - **api-contacts.spec.ts**: Handles contacts CRUD and error scenarios.
   - **api-likes.spec.ts**: Validates likes functionality on transactions.
   - **api-notifications.spec.ts**: Tests receiving, creating, updating, and error handling for notifications.
   - **api-testdata.spec.ts**: Provides coverage for test data APIs for users, contacts, bank accounts etc.
   - **api-transactions.spec.ts**: Comprehensive coverage for payments, requests, status updates, and filtering.
   - **api-users.spec.ts**: Tests user endpoints—CRUD, authentication, profile queries, and searches.

4. **Setup and Environment**

   - Each suite typically:
     - Seeds the database to a known state before running.
     - Authenticates a Cypress test user (using API commands for efficiency and control).
     - Utilizes test data generation (e.g., [Faker](https://fakerjs.dev/)) where dynamic input is required.
     - Asserts not just HTTP status codes but also response structures, authorization, and entity associations.

## Useful details

- **Test Execution**: Cypress tests are designed to run in a clean, deterministic environment, relying heavily on database seeding and controlled authentication to ensure reproducibility.
- **Custom Cypress Commands**:
  - `cy.request()`: Main method for sending HTTP requests to API endpoints.
  - `cy.task()`: Runs tasks in the Node.js process, often for manipulating the database.
  - `cy.database()`: Abstraction for specific DB seeding or queries (implementation-dependent).
  - `cy.loginByApi()`: Convenience method for programmatic login.
- **Error Handling**: Negative test cases are included to cover scenarios such as invalid input, forbidden actions, and invalid states. Proper HTTP codes (e.g., 422, 401, 404) and response payload validation are asserted.
- **No Fixtures**: Most suites interact directly with the database and APIs rather than relying on static fixtures, ensuring tests reflect real-world flows.
- **Extensibility**: Add new `.spec.ts` files alongside existing files to expand coverage for additional API domains or features.
- **Debugging**: When a test fails, Cypress provides detailed stack traces, network request logs, and time-travel debugging in the Test Runner UI.
- **Continuous Integration**: These suites are designed to be part of your project’s CI/CD pipeline to automatically catch API regressions on every code push.

For further details on extending these tests, custom command implementations, or integrating with your CI setup, refer to the main project documentation or the Cypress documentation as required.
