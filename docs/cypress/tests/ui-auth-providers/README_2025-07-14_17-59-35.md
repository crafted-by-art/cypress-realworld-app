## Overview
This folder contains the suite of Cypress end-to-end (E2E) tests responsible for validating user authentication and onboarding flows via various third-party identity providers (IdPs). The primary goal is to ensure seamless and reliable integration with external services like Google, Okta, Auth0, and AWS Cognito.

Each test suite simulates a critical user journey: logging in, completing the initial application setup (e.g., creating a bank account), and logging out. By testing these core paths, this suite acts as a crucial quality gate, guaranteeing that fundamental user-facing functionality remains operational across different authentication methods.

## Usage
These tests are designed to be executed by the Cypress test runner. To run them, ensure the project is set up with all its dependencies installed.

**Prerequisites:**
1.  A running instance of the application backend and database.
2.  Properly configured Cypress environment variables (`cypress.env.json` or system-level) for the credentials of the authentication provider you wish to test. The tests are written to run conditionally based on the presence of these variables.

**Execution:**
You can run the tests using the Cypress Test Runner or via the command line.

*   **To open the Test Runner UI:**
    ```bash
    npx cypress open
    ```
    Then, select the desired spec file from the `ui-auth-providers` directory.

*   **To run all tests in this suite headlessly:**
    ```bash
    npx cypress run --spec "cypress/tests/ui-auth-providers/*.spec.ts"
    ```

## Useful details
The tests rely on common patterns, such as using `cy.task("db:seed")` to ensure a consistent database state before each run and `cy.intercept()` to reliably wait for asynchronous API calls.

The structure of this directory is organized by authentication provider:

*   **`auth0.spec.ts`**:
    This test validates the complete user lifecycle with Auth0. It focuses on a programmatic login to speed up execution, verifies that the user session is correctly cached between tests, and confirms the user can successfully complete the multi-step onboarding wizard.

*   **`cognito.spec.ts`**:
    This suite covers authentication via AWS Cognito. It is uniquely designed with two conditional test flows: one for a fast, programmatic API-based login and another for a full UI-based login using `cy.origin()`, which provides more comprehensive coverage of the user's browser experience.

*   **`google.spec.ts`**:
    This test ensures that users can authenticate via Google and complete the subsequent onboarding. It uses a custom `cy.loginByGoogleApi()` command to perform a programmatic login, focusing the test's scope on the post-authentication user journey within our application.

*   **`okta.spec.ts`**:
    This file provides two distinct validation paths for Okta integration. The primary flow tests the full "happy path" for a new user: programmatic login, onboarding, and logout. An alternative flow tests the standard UI login and verifies that a newly created user is presented with the correct empty-state messages, confirming the initial UI is rendered as expected.
