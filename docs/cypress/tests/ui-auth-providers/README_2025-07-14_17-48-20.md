## Overview

The `ui-auth-providers` folder contains Cypress end-to-end test suites specifically designed to validate authentication and onboarding flows against multiple popular identity providers (Auth0, AWS Cognito, Google, and Okta). Each test suite simulates a real user journey: logging in through the provider, completing onboarding activities (such as filling out bank account information), confirming successful authentication and onboarding, and testing log-out behavior. The tests also ensure consistent experience across different device form-factors (desktop vs. mobile). This folder's primary purpose is to guarantee that the application's authentication workflows are robust, correctly integrated with these providers, and deliver a seamless onboarding experience for users.

## Usage

To use these Cypress test suites in your project:

1. **Setup Prerequisites:**  
   Ensure your environment is configured with the necessary provider credentials (e.g., Auth0, AWS Cognito, Google, Okta). Test secrets and environment variables must be present, and the application under test should be running and accessible to the Cypress test runner.

2. **Running Tests:**  
   In your project root (with Cypress and dependencies installed), execute:
   ```sh
   npx cypress open
   ```
   or run headless via:
   ```sh
   npx cypress run --spec "cypress/tests/ui-auth-providers/*.spec.ts"
   ```

3. **Test Structure:**  
   - Each `.spec.ts` file corresponds to a provider:
     - `auth0.spec.ts.md` — Auth0 login & onboarding
     - `cognito.spec.ts.md` — AWS Cognito authentication flows
     - `google.spec.ts.md` — Google login automation
     - `okta.spec.ts.md` — Okta authentication via API/UI
   - Each suite seeds the database, intercepts external/service calls, simulates user login via the provider, completes onboarding flow, and asserts log-out and post-auth behaviors.
   - Responsive testing: The tests adapt to mobile and desktop views, ensuring full coverage.

4. **Key Custom Commands:**  
   The tests rely on Cypress custom commands for login (e.g., `cy.loginByOktaApi`, `cy.loginByCognitoApi`, `cy.loginToA`) and onboarding simulation.

## Useful details

- **Seeding and Intercepts:**  
  Each suite ensures a clean state by seeding the database before tests and uses `cy.intercept` to monitor or mock network requests, particularly for GraphQL APIs and account creation steps.

- **Provider-specific Flows:**  
  - **Auth0:** Simulates login, onboarding, verifies transaction list, and tests logout; caches logins to speed up subsequent runs.
  - **AWS Cognito:** Supports both programmatic (`cy.loginByCognitoApi`) and UI (`cy.loginByCognito`) login methods.
  - **Google:** Tests run only if a Google client ID is configured, using a custom Google login utility.
  - **Okta:** Supports login via API and UI; checks for onboarding flows and the absence of user-specific data post-logout.

- **Mobile/Desktop Handling:**  
  The test logic adapts based on the detected viewport, confirming that onboarding and logout behaviors work as expected for all users, regardless of device.

- **Extensibility:**  
  New authentication providers can be tested by modeling new specs after these files—seeding, intercepting, logging in via the provider, simulating onboarding, and validating logout.

- **Isolation:**  
  No external fixtures are loaded; each test internally manages state to ensure test repeatability and reliability.

- **Recommended for:**  
  - Continuous Integration (CI) to guarantee authentication integration stability.
  - Regression testing before deployments when updating authentication logic or providers.
  - Developers integrating new auth providers.

In summary, the `ui-auth-providers` folder is a critical piece of quality assurance infrastructure, automating verification of authentication and onboarding workflows for multiple SSO providers within your application ecosystem.
