## Overview
This directory houses a suite of Cypress end-to-end (E2E) tests focused on validating user authentication and onboarding flows via various third-party identity providers. The primary purpose of these tests is to simulate and verify the complete initial user journey—from login, through multi-step setup, to logout—for each integrated authentication service.

In the context of the larger project, this folder is critical for maintaining the reliability of our core user acquisition funnels. These tests act as a quality gate, ensuring that integrations with external providers like **Auth0, AWS Cognito, Google, and Okta** remain stable and functional, preventing regressions in mission-critical user-facing features.

## Usage
The tests within this folder are designed to be run using the standard Cypress test runner. You can execute them individually or as part of the entire test suite.

To run all tests:
```bash
npx cypress run
```

To open the interactive Cypress Test Runner and select a specific test file:
```bash
npx cypress open
```

Execution of certain tests or test modes is controlled by environment variables. This is necessary to provide credentials securely and to toggle specific testing strategies.

For example, to run a test that depends on a specific environment variable:
```bash
# The Google test is skipped unless this variable is present
CYPRESS_googleClientId="YOUR_GOOGLE_CLIENT_ID" npx cypress run --spec "cypress/e2e/ui-auth-providers/google.spec.ts"

# The Okta/Cognito tests may use a flag to switch between UI and programmatic login
CYPRESS_LOGIN_MODE="API" npx cypress run --spec "cypress/e2e/ui-auth-providers/okta.spec.ts"
```

Consult the individual test files for details on required environment variables.

## Useful details
*   **Conditional Execution**: Some tests, such as `google.spec.ts`, are designed to be skipped by default. They will only execute if the required environment variables (e.g., `googleClientId`) are provided. This design prevents test failures in local or CI environments where secrets for all providers may not be configured.

*   **Dual-Mode Login Flows**: The tests for Okta and Cognito are built with a powerful dual-mode capability. They can be configured to run either a full UI-based login or a faster, programmatic (API-based) login.
    *   **UI Mode**: Simulates a real user's actions, providing a comprehensive validation of the login experience.
    *   **Programmatic Mode**: Bypasses the UI to obtain an authentication session via API calls. This is significantly faster and is ideal for setting up an authenticated state required by other E2E tests, thereby optimizing overall test suite execution time.

*   **Provider-Centric Structure**: Each test file is self-contained and dedicated to a single authentication provider. This modular structure makes it easy to isolate, manage, and debug tests for a specific integration without impacting others.
