## Overview
This directory contains demonstration and placeholder end-to-end (E2E) tests designed specifically to showcase the capabilities of the Cypress Studio feature. Its primary purpose is to serve as a starting point and educational tool for team members, enabling them to rapidly generate new tests by recording user interactions directly in the browser.

In the context of the broader project, this folder provides a practical template for "low-code" or "no-code" test creation. It establishes a best-practice setup where complex preconditions, such as database state and user authentication, are handled programmatically, allowing the test author to focus solely on recording the specific user flow under test.

## Usage
These tests are intended to be used with the Cypress Test Runner and the Cypress Studio feature. The tests themselves are empty stubs; you will populate them by recording actions.

To use this test suite:
1.  Launch the Cypress Test Runner and open the E2E testing environment.
2.  Select the `cypress-studio.spec.ts` test file from the list.
3.  Hover over a test name (e.g., `it('creates a new transaction', ...)`).
4.  Click the "magic wand" icon to the right of the test name to launch Cypress Studio for that test.
5.  The `beforeEach` hook will automatically run, seeding the database and logging you in as a test user.
6.  Interact with the application's user interface to perform the desired actions for the test case.
7.  Once finished, click "Save Commands" in the Cypress Studio command log.
8.  The recorded steps will be automatically written into the test file, completing the test case.

## Useful details
The core of this demonstration suite lies in its `beforeEach` hook, which ensures a consistent and clean state for every test run. Understanding its components is key to leveraging this pattern effectively across the project.

*   **Database Seeding (`cy.task("db:seed")`):** Before each test, the database is reset to a known, predictable state. This is a critical practice for test isolation, ensuring that the outcome of one test does not affect another. It guarantees that every test starts from the same clean slate.

*   **Programmatic Login (`cy.login()`):** After the database is seeded, the test suite programmatically logs in a user. This is achieved by using a custom Cypress command that likely interacts with the backend directly or sets authentication tokens in the browser. This approach is significantly faster and more reliable than repeatedly scripting UI interactions with the login form for every test. It separates the concern of "being logged in" from the actual user behavior being tested.

*   **Placeholder Tests:** The individual `it(...)` blocks are intentionally left empty. They serve as named containers that you will populate by recording interactions with Cypress Studio, as described in the **Usage** section.
