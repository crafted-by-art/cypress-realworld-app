## Overview

The `auth-provider-commands` folder is a centralized library for Cypress custom commands that facilitate authentication workflows against multiple popular identity providers (IdPs) in automated end-to-end tests. By abstracting the complexities of logging in via Auth0, Amazon Cognito, and Okta, this module enables fast and reliable test setups for applications using these authentication platforms. Its main objective is to simplify and standardize test authentication, whether via direct API calls or browser-based UI automation, thus supporting robust and maintainable test suites for modern web applications that depend on external authentication providers.

This folder plays a crucial role in ensuring that your Cypress tests can programmatically log in users, bypassing manual UI actions when possible while still supporting interactive flows as a fallback, and handling session caching for improved test performance.

---

## Usage

1. **Prerequisites**  
   - Cypress must be installed and properly configured in your project.
   - The `auth-provider-commands` folder must be placed under `cypress/support`.
   - Each authentication command may require specific environment variables (such as usernames, passwords, or API endpoints), set in your Cypress configuration.

2. **Importing the Command Files**  
   In your Cypress support file (e.g., `cypress/support/commands.js` or `cypress/support/e2e.js`):
   ```js
   import './auth-provider-commands/auth0';
   import './auth-provider-commands/cognito';
   import './auth-provider-commands/okta';
   ```

3. **Usage in Tests**  
   Call the relevant Cypress custom command within your test `beforeEach` or setup block:
   
   - **Auth0**
     ```js
     cy.loginToAuth0('user@example.com', 'password123');
     ```
   
   - **Amazon Cognito (API or UI)**
     ```js
     cy.loginByCognitoApi('user@example.com', 'password123');
     // or
     cy.loginByCognito('user@example.com', 'password123');
     ```
   
   - **Okta (API or UI)**
     ```js
     cy.loginByOktaApi('user@example.com', 'password123');
     // or
     cy.loginByOkta('user@example.com', 'password123');
     ```

   These commands will handle logging in the user and establishing a valid authenticated session within the Cypress test context.

4. **Session Caching**  
   Where supported, these commands utilize Cypress's `cy.session` API to cache session state and avoid redundant authentication in each test, speeding up your test suite.

## Useful details

- **Strategies Supported:**
  - **API-based Authentication:** For Cognito and Okta, the API approach fetches and injects tokens, making logins fast and stable (when supported by your app's architecture).
  - **UI-based Authentication:** For all providers, a UI automation fallback logs in through the real login page to mimic authentic user behavior or test SSO flows.

- **Session Validation:** Authentication success is asserted by checking expected tokens or state (e.g., `localStorage` items like `authState`).

- **Security and Maintenance:** Do not commit plaintext credentials. Use Cypress environment variables for sensitive data. Update selectors and flows promptly if the IdP login pages change.

- **Extensible Design:** The modular per-provider structure makes it easy to add support for new authentication providers or tweak existing flows without affecting unrelated code.

- **Debugging:** Each command logs diagnostic steps to the Cypress command log, aiding in test authoring and troubleshooting authentication failures.

- **Integration:** These commands are ideal for CI/CD test pipelines, ensuring reliable setup for all authenticated test cases.
