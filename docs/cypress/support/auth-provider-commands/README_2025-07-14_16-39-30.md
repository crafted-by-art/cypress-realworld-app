## Overview
This folder centralizes custom Cypress commands designed to handle user authentication with various third-party identity providers. The primary purpose is to abstract away the complex and provider-specific login flows, offering a standardized and reusable set of commands for end-to-end tests. By doing so, it significantly simplifies the process of writing tests that require an authenticated user state.

The commands within this directory support multiple authentication strategies, including fast, programmatic API-based logins and more realistic UI-driven logins. This modular approach makes the test suite more maintainable, robust, and easier to extend as new authentication providers are introduced. The supported providers include **Auth0**, **AWS Cognito**, and **Okta**.

## Usage
The commands in this folder are intended to be imported into your main Cypress support file (e.g., `cypress/support/e2e.ts`). Once imported, they become available globally under the `cy` object in all of your test files.

1.  **Import the required provider commands:**
    In your `cypress/support/e2e.ts` file, add an import statement for the authentication provider your project uses.

    ```typescript
    // cypress/support/e2e.ts

    // Import commands for your specific auth provider
    import './auth-provider-commands/cognito';
    // or
    // import './auth-provider-commands/auth0';
    // or
    // import './auth-provider-commands/okta';
    ```

2.  **Utilize the command in a test:**
    Within your test files, you can now use the custom login command, typically inside a `beforeEach` hook, to ensure the test runs with an authenticated user session.

    ```typescript
    // cypress/e2e/my-feature.cy.ts

    describe('My Protected Feature', () => {
      beforeEach(() => {
        // Use the appropriate command for your provider.
        // This example uses an API-based Cognito login for speed.
        cy.loginByCognitoApi(Cypress.env('TEST_USER_EMAIL'), Cypress.env('TEST_USER_PASSWORD'));

        // The session is now active, and you can visit protected pages.
        cy.visit('/dashboard');
      });

      it('should display user-specific data on the dashboard', () => {
        cy.get('[data-cy="user-greeting"]').should('be.visible');
        // ... more assertions
      });
    });
    ```

For detailed usage, prerequisites, and specific method signatures for each provider, please refer to the documentation within the individual files:
*   `auth0.ts`
*   `cognito.ts`
*   `okta.ts`

## Useful details
*   **Authentication Strategies:** Most providers in this folder offer two login strategies:
    *   **API-based login (e.g., `loginByCognitoApi`)**: This is the recommended approach for most tests as it's significantly faster and more reliable, bypassing the UI entirely. It's ideal for `beforeEach` hooks.
    *   **UI-based login (e.g., `loginByCognito`)**: This method simulates a real user typing credentials and clicking buttons. It should be used sparingly, primarily for tests that specifically validate the login page's user interface and functionality.

*   **Extensibility:** This directory is designed to be easily extensible. To support a new identity provider, simply create a new file (e.g., `my-new-provider.ts`) that follows the existing pattern of defining and exporting one or more custom Cypress commands.

*   **Configuration:** Sensitive information like usernames, passwords, and client IDs should be managed through Cypress environment variables (e.g., in `cypress.env.json`) rather than being hardcoded in tests. The individual command files provide details on the required environment variables.
