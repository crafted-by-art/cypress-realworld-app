## Overview
This directory contains a collection of custom Cypress commands designed to handle authentication with various third-party identity providers (IdPs). The primary purpose of these commands is to abstract and streamline the login process within end-to-end tests, making them more robust, faster, and easier to maintain.

By centralizing the authentication logic for providers like **AWS Cognito**, **Auth0**, and **Okta**, we avoid repetitive login steps in our test specs. The commands leverage powerful Cypress features like `cy.session` to cache login states across tests and `cy.origin` to seamlessly interact with external authentication domains.

Two primary authentication strategies are implemented for each provider where applicable:
1.  **UI-Based Login:** Simulates a real user logging in through the provider's web interface. This is useful for validating the login flow itself.
2.  **API-Based (Programmatic) Login:** Bypasses the UI entirely by interacting directly with the provider's API to fetch authentication tokens. This method is significantly faster and more reliable, making it the preferred choice for tests that simply require a pre-authenticated state.

## Usage
These commands are intended to be used as prerequisites in `beforeEach` hooks or at the beginning of individual tests that require a user to be logged in. To make them globally available, they should be imported into the `cypress/support/e2e.ts` (or equivalent command) file.

**Example: Using a command in a test spec**

Let's assume you need to test a feature on a dashboard that is only accessible to authenticated users. Instead of manually logging in for every test, you can call the appropriate custom command.

```typescript
// in cypress/specs/dashboard.cy.ts

describe('Dashboard Feature', () => {
  beforeEach(() => {
    // This command handles the entire login flow and session caching.
    // We use Cypress environment variables to avoid hardcoding credentials.
    cy.loginByOktaApi(
      Cypress.env('OKTA_USERNAME'),
      Cypress.env('OKTA_PASSWORD')
    );
    
    // The application will be in a logged-in state for every test in this block.
    cy.visit('/dashboard');
  });

  it('should display the user welcome message', () => {
    cy.contains('h1', 'Welcome, User!').should('be.visible');
  });

  it('should allow access to protected data', () => {
    cy.get('[data-cy="protected-data-table"]').should('exist');
  });
});
```

## Useful details
*   **Choosing the Right Command:**
    *   Use the **API-based commands** (e.g., `loginByCognitoApi`, `loginByOktaApi`) for the vast majority of tests. They are faster and less prone to flakiness from UI changes on the provider's login page.
    *   Use the **UI-based commands** (e.g., `loginToAuth0`, `loginByCognito`, `loginByOkta`) sparingly, specifically for tests designed to verify the user-facing login journey itself.

*   **Session Caching with `cy.session`:** All the UI-based login commands are wrapped in `cy.session`. This means Cypress will perform the full login flow only once per spec file. For subsequent tests, it will restore the session (cookies, `localStorage`, etc.) from cache, providing a significant performance boost.

*   **Credential Management:** Sensitive data like usernames, passwords, and API keys should **never** be hardcoded. Store them securely in `cypress.env.json` or as system environment variables, and access them in your tests via `Cypress.env('YOUR_VARIABLE_NAME')`.

*   **Cross-Origin Interaction:** The `cy.origin` command is critical for interacting with the IdP's domain, which is different from the application's domain (`localhost`). This command creates a secure context to execute commands like `cy.get()` and `cy.click()` on the external login page.
