## Overview
This directory, `cypress/support/`, is a foundational component of our Cypress testing framework. Its primary purpose is to house and configure reusable behaviors, custom commands, and global setups that are available across all test files. By centralizing this logic, we maintain clean, readable, and DRY (Don't Repeat Yourself) test suites.

This folder is automatically processed by Cypress before executing tests, making it the ideal location for extending Cypress's core functionality. It contains separate configurations for End-to-End (`e2e.ts`) and Component (`component.ts`) tests, as well as a modular system for handling authentication with various providers located in the `auth-provider-commands` sub-directory.

## Usage
The contents of this folder are largely configured to be used automatically by the Cypress test runner. You do not need to import these files into your individual test specs (`*.cy.ts`).

*   **For E2E Tests**: Cypress automatically includes `e2e.ts` before running each E2E spec file. To make a new command or utility available to all E2E tests, simply import it into `e2e.ts`.
*   **For Component Tests**: Similarly, `component.ts` is automatically included before each component spec. This is where you would import commands or mounting utilities specific to component testing.

**Example: Adding a new custom command**
1.  Define the command's implementation in a relevant file, such as `commands.ts` or a more specific file within `auth-provider-commands/`.
2.  Add the TypeScript definition for the command to the `Cypress.Chainable` interface in `cypress/support/index.d.ts` (or a similar typing file) to get IntelliSense.
3.  Import the command file into `e2e.ts` (for E2E tests) or `component.ts` (for component tests).
4.  Use the new command in any test spec file: `cy.myNewCommand()`.

## Useful details
This section breaks down the structure and purpose of the key files and folders within `cypress/support/`.

*   **`e2e.ts`**: The main entry point for all End-to-End tests. It is executed before any spec file runs. Use this file to import all necessary custom commands and to define global `beforeEach` or `afterEach` hooks that should apply to all E2E tests.

*   **`commands.ts`**: The primary container for custom Cypress commands. Functions defined here extend the global `cy` object, allowing for the creation of powerful, domain-specific actions (e.g., `cy.login()`, `cy.createPost()`). This abstracts complex sequences of actions into single, readable commands.

*   **`component.ts`**: The entry point for all Component tests. It's used to import commands and configurations specific to testing components in isolation. It also typically contains the setup for mounting components using the `mount` command from `@cypress/react` or `@cypress/vue`.

*   **`utils.ts`**: A collection of general-purpose helper functions that are not Cypress commands. These utilities can be imported into command files or test specs as needed. This helps keep the `cy` namespace clean and separates command logic from general utility logic (e.g., data generation, string formatting).

*   **`component-index.html`**: A template HTML file that Cypress uses as a test harness for Component Tests. When you mount a component, Cypress renders it within this file, allowing you to style the test environment or add global HTML elements if needed.

*   **`auth-provider-commands/`**: This sub-directory modularizes authentication logic. By creating a separate file for each identity provider (e.g., `auth0.ts`, `cognito.ts`), we can easily manage, update, or add new authentication flows without impacting the rest of the test suite. This design promotes scalability and maintainability. See the [auth-provider-commands README](./auth-provider-commands/README_2025-07-14_16-39-30.md) for more details.
