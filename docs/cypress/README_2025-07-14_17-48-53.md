## Overview

The `cypress` folder is the main directory holding all Cypress-based end-to-end (E2E), integration, and API testing resources for this project. It acts as the root namespace for organizing, configuring, and extending the Cypress testing environment, ensuring automated validation of both frontend and backend components throughout the software development lifecycle.

Within this folder, you will find:

- **Configuration files** that define global test behavior and settings.
- **TypeScript definitions** that provide enhanced type safety for tests.
- **Support infrastructure** that supplies reusable commands, utilities, and authentication flows.
- **A rich hierarchy of tests**—spanning UI, API, authentication flows, and demo scenarios—organized into discoverable subfolders.

By centralizing these components, the `cypress` folder enables robust, maintainable, and scalable automated testing, which is essential for delivering high-quality software and supporting continuous integration pipelines.

## Usage

To utilize the Cypress testing suite provided here:

1. **Install Project Dependencies:**  
   Ensure you have all required dependencies installed. Typically, running `npm install` or `yarn install` from your project root will accomplish this.

2. **Run Cypress Tests:**
   - **E2E and UI Tests:**  
     Open the Cypress test runner with `npx cypress open` or run the tests headlessly in your command line using `npx cypress run`.
   - **API & Component Tests:**  
     Tests in the `tests` folder are automatically discovered and executed via standard Cypress commands.

3. **Leverage Test Utilities:**
   - Reuse and extend custom commands and utilities from the `support` folder.
   - Reference and, if needed, augment authentication helpers to test flows involving providers such as Auth0, Cognito, Okta, and Google.

4. **Structure New Tests Appropriately:**  
   Add new test cases or suites to the clearly defined subfolders (`api`, `ui`, `ui-auth-providers`, `demo`) within `tests`, ensuring consistency and discoverability.

## Useful details

- **Folder Structure:**
  - `cypress.config.ts` — Configures global Cypress options, e.g., base URLs, reporter settings.
  - `global.d.ts` — TypeScript declarations to augment or type-check custom commands or environmental variables.
  - `support/` — Houses custom commands (`commands.ts`), component testing support (`component.ts`, `component-index.html`), initialization logic for tests (`e2e.ts`), utility helpers (`utils.ts`), and authentication command providers (`auth-provider-commands/`).
  - `tests/` — Contains all actual test suites, split into subfolders for `api`, `ui`, provider-based auth flows, and demos.
- **Extensibility:**  
  The support infrastructure encourages sharing and centralizing helpers, reducing test duplication, and simplifying maintenance.
- **Documentation:**  
  Each significant subfolder (e.g., `support`, `tests/api`, `tests/ui`, and each auth provider support folder) includes a detailed README explaining its specific role and usage patterns.
- **Authentication:**  
  Ready-to-use authentication commands and helpers for major providers accelerate the creation of secure E2E scenarios.

**Tip:** Before contributing new or modified tests, review the corresponding README under each subfolder for best practices and to avoid redundancy.

Cypress empowers rapid, confident development by integrating automated testing directly in your build workflow. The structure presented here makes it easy for teams to enhance and scale test coverage as the application grows.
