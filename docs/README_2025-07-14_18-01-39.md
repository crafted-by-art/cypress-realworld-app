## Overview
This `docs` directory serves as the central knowledge base and technical documentation hub for the entire project. Its primary purpose is to provide clear, developer-focused documentation that explains the architecture, configuration, and implementation details of the project's most critical components.

The documentation within this folder is meticulously structured to mirror the project's source code layout. This approach ensures that developers can easily find context and high-level explanations for any given file or module. The core focus of the documentation is on the project's robust testing infrastructure (powered by Cypress) and its automated Continuous Integration pipeline (managed by CircleCI), which are fundamental to our quality assurance and deployment strategies.

In essence, this directory is the single source of truth for understanding *how* our application is built, tested, and maintained. It is an indispensable resource for onboarding new team members, facilitating code reviews, and enabling long-term project maintainability.

## Usage
To use this documentation, simply navigate the directory structure as you would the project's source code. If you want to understand the purpose and functionality of a specific file, such as `cypress/support/commands.ts`, you can find its corresponding documentation at `docs/cypress/support/commands.ts.md`.

The documentation is organized into logical sections:

*   **`./.circleci/`**: Contains documentation for our CircleCI configuration. Refer to this section to understand our continuous integration workflows, including how tests are automatically triggered and executed.
*   **`./cypress/`**: Houses all documentation related to our Cypress testing suite. This is the most extensive section, covering everything from custom commands and support utilities to the implementation details of our API, UI, and component tests.
*   **Root-level Files (`*.md`)**: Documentation for top-level configuration files, such as `vite.config.ts.md` and `cypress.config.ts.md`. These documents explain the setup of our build tools and test runner.

## Useful details
*   **Living Documentation:** This documentation is intended to be a "living" resource, updated in tandem with the source code to ensure it remains accurate and relevant.
*   **High-Level Focus:** While the code itself contains inline comments for implementation specifics, the documents here focus on the "why"—the high-level purpose of a component and its role within the broader system.
*   **Key to Onboarding:** New developers should consider this directory their starting point for gaining a comprehensive understanding of the project's technical landscape before diving deep into the codebase.
